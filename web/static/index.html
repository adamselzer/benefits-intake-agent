    An intake agent that reads a messy benefits application, checks it against the rules, and hands a caseworker a recommendation — without ever deciding eligibility itself, and without ever denying anyone.

    Who it is for, and the need

    The user is a benefits caseworker screening SNAP (food assistance) applications. Each application arrives as a form plus uploaded documents — pay stubs, leases, utility bills, IDs — that are often incomplete, inconsistent, or simply wrong. Screening them by hand is slow. Automating them naively is dangerous: a single misread number can become a wrongful denial of food assistance for a family that qualifies.

    So the need is narrow and specific. Do the tedious reading and cross-checking, surface exactly what a human must resolve, and stop short of the decision. The dangerous error here is not a false denial (the agent cannot deny) but a case quietly waved through that should have reached a person. Everything below serves keeping that number at zero.

    What it does

    1. Ingest — validate the stated application.
    2. Extract — read each document with the model and pull structured fields.
    3. Validate — cross-check documents against the stated application, surface conflicts, flag anything that could not be verified.
    4. Screen — hand the facts to a deterministic rules engine that returns an eligibility determination with a rule trace and policy citations.
    5. Route — clear-eligible, clear-ineligible, or needs-human-review. There is no "denied".

    Technical decisions

    The eligibility test is a tool call, not model reasoning

    The screen step calls a separate, deterministic rules-as-code engine that returns a determination with citations. The model orchestrates; tested, auditable code decides.

    Instead of: letting the model reason about eligibility from policy text. In government, "the model said so" is not a basis for denying benefits.

    A LangGraph pipeline, so the boundary is inspectable

    Each step is a named node with explicit edges, which makes the point where control leaves the model and enters deterministic code a thing you can see, not a claim in a README.

    Instead of: one opaque prompt that does extraction, screening, and routing at once.

    The no-deny invariant is structural

    The route type has three values and none of them is a denial. A test asserts no case ever produces one. Any conflict, low-confidence financial field, or missing document forces human review.

    Instead of: a prompt instruction telling the model not to deny. Prompts drift; a type and a test do not.

    Composition over MCP

    The rules engine is a separate project exposed over the Model Context Protocol. This app calls it directly for reliability, and ships a demo of the same screen performed over MCP — the deployable shape, where the rules core versions and deploys on its own.

    Instead of: copying the eligibility logic into the agent, where it would drift out of sync with the source of truth.

    Extraction is injectable; evaluation comes first

    The document reader is a dependency: Claude vision in production, a deterministic stand-in for tests and offline runs. That lets the safety logic be verified without a key, and the eval measures the metric that maps to harm — the rate at which a should-review case is waved through.

    Instead of: hard-wiring the model call into the graph and demoing it once by hand.

    Design decisions

    The boundary is the layout

    The recommendation puts "what the model read" beside "what the rules decided". The single most important idea in the system — where probabilistic reading ends and deterministic deciding begins — is the structure of the screen, not a sentence.

    Instead of: a flat list of fields that hides which were read by a model and which were decided by code.

    Provenance is a first-class element

    Every value carries a source tag — document, rule, or stated-but-unverified — and IDs and citations are set in monospace, like audit artifacts. A caseworker (or an auditor) can trace any claim to where it came from.

    Instead of: presenting model output and policy output as if they were the same kind of fact.

    Cut the telemetry that carried no signal

    An earlier version drew a confidence bar for every extracted field. On clean inputs every bar was full and identical — a rigor-shaped decoration. It was removed. The emphasis belongs on the exception: the unverified field that routes a case to a human.

    Instead of: keeping a metric on screen because it looks thorough. A uniform "all good" indicator is noise at best and dishonest at worst.

    A bespoke front-end where it earns it

    This surface is hand-built rather than a themed framework default, because the layout, the provenance system, and the restraint are the work. The portfolio's other projects stay on a lighter framework where that craft would not pay off.

    Instead of: four identical framework apps. Quality where it matters beats uniformity.

    Honest limitations

    • The documents are synthetic and clean, so extraction reads them perfectly. The confidence-and-escalate machinery exists for the real-world case (photographed, skewed, partial) that this set does not yet stress.
    • Recovery-under-injection is measured with the deterministic extractor; the injection corrupts a field the vision model never reads, so testing it under vision needs a re-rendered document.
    • Under real vision, a case takes about 44 seconds, dominated by sequential document reads. Batching would cut that sharply.
